UPGRADE: v6.8.0 TO v6.8.1 REUSABLE SCHOOLS EDITION

PURPOSE
This upgrade restricts submission to the assigned class teacher, restricts publication to the assigned class teacher and System Administrator, and adds class-level bulk submit, approve, and publish actions.

BEFORE DEPLOYMENT
1. Preserve the confirmed working v6.8.0 ZIP in a secure rollback location.
2. Create and verify an encrypted system backup.
3. Download or separately preserve the current production GITHUB_PAGES_FRONTEND/config.js.
4. Confirm that every class has the correct official class_teacher_id assignment.
5. Confirm that subject teachers have active class_subjects assignments for only their subjects.

DATABASE UPGRADE
1. Open the Supabase SQL Editor for the production project.
2. Run 07_schema.sql only.
3. Confirm that the final result is: 07 SCHEMA: PASS.
4. Do not rerun historical hotfix files.

FRONTEND DEPLOYMENT
1. Replace the deployed frontend files with the contents of the v6.8.1 GITHUB_PAGES_FRONTEND folder.
2. Preserve the real Supabase Project URL and Publishable key in config.js.
3. Confirm that service-worker.js contains the v6.8.1 cache name.
4. Reload the site and allow the new service worker to activate. Use a hard refresh if an older interface remains cached.

EDGE FUNCTION
Redeploy scheduled-backup/index.ts so newly generated backup metadata records schema version 6.8.1. No backup table list change is introduced by this release.

REQUIRED ACCEPTANCE TESTS

A. SUBJECT TEACHER
1. Sign in as a subject teacher assigned to one subject in a test class.
2. Open a draft report in that assigned class and enter or update only the assigned subject scores.
3. Confirm that the report saves normally.
4. Confirm that Submit for Principal approval is not displayed.
5. Confirm that Publish report is not displayed for an approved report.
6. Confirm that Submit class reports and Publish class reports are not displayed.
7. Attempt direct RPC calls to transition_report_status for submitted and published, if using a test client. Confirm both are denied.

B. CLASS TEACHER INDIVIDUAL SUBMISSION
1. Sign in as the officially assigned class teacher.
2. Open a complete draft report in the teacher's home class.
3. Confirm that Submit for Principal approval is displayed and succeeds.
4. Open an incomplete report and confirm submission is rejected until all assigned subjects and required scores are complete.
5. Open a report from a class where the same teacher is only a subject teacher. Confirm submission is unavailable or denied.

C. CLASS TEACHER BULK SUBMISSION
1. On Report Cards, select one specific term and the teacher's official home class.
2. Select Submit class reports.
3. Confirm that complete draft or returned reports become submitted.
4. Confirm that incomplete reports remain unchanged and appear in the result summary.
5. Confirm that students without a report are counted in the summary.
6. Select a class where the teacher is only a subject teacher and confirm the bulk action is blocked.

D. PRINCIPAL APPROVAL
1. Sign in as the Principal.
2. Confirm that individual submitted reports can be approved.
3. Select one class and term, then select Approve class reports.
4. Confirm that submitted or class-reviewed reports become approved.
5. Confirm that the Principal does not see Publish report or Publish class reports.
6. Confirm that report-return controls still work for corrections.

E. CLASS TEACHER PUBLICATION
1. Sign in as the assigned class teacher after Principal approval.
2. Confirm that an approved report can be published individually.
3. Select the teacher's home class and term, then select Publish class reports.
4. Confirm that approved reports become published.
5. Confirm that official PDF generation runs for each newly published report.
6. Confirm that published reports remain accessible through the normal report and guardian workflows.
7. Confirm that the teacher cannot publish a class where the teacher is only a subject teacher.

F. SYSTEM ADMINISTRATOR PUBLICATION
1. Sign in as the System Administrator.
2. Confirm that Submit for Principal approval and Submit class reports are not available.
3. Confirm that approved reports can be published individually.
4. Select any class and term, then select Publish class reports.
5. Confirm that approved reports are published and official PDFs are stored.

G. AUDIT AND REGRESSION
1. Verify report_workflow_events for each bulk-transitioned report.
2. Verify report_revisions and actor identifiers.
3. Verify report_publications for newly published reports.
4. Verify workflow notifications.
5. Recheck score entry, attendance, promotion, alphabetical directories, teacher class visibility, PDF download, QR verification, and encrypted backup.

ROLLBACK
If acceptance fails, stop further report workflow actions, preserve evidence and logs, restore the verified pre-upgrade database backup, and redeploy the preserved v6.8.0 frontend and scheduled-backup function. Do not attempt an unverified partial rollback on production data.
