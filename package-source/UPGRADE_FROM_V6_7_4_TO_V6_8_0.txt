UPGRADE: v6.7.4 TO v6.8.0 REUSABLE SCHOOLS EDITION

PROTECTED BASELINE
1. Keep the confirmed working v6.7.4 ZIP unchanged as the rollback baseline.
2. Create and verify a current encrypted system backup before upgrading.
3. Export or record the current config.js Project URL and Publishable key.

DATABASE UPGRADE
1. Open the existing school's Supabase SQL Editor.
2. Run 06_schema.sql once.
3. Confirm the transaction completes without an error.
4. Do not rerun files 01 through 05 for this upgrade.

EDGE FUNCTION UPGRADE
1. Redeploy supabase/functions/scheduled-backup/index.ts.
2. The other Edge Functions are unchanged, but may be redeployed with the package if desired.
3. Verify NIS_BACKUP_ENCRYPTION_KEY and NIS_CRON_SECRET remain configured.

FRONTEND UPGRADE
1. Preserve the existing production config.js values.
2. Replace the deployed GitHub Pages frontend files with the contents of GITHUB_PAGES_FRONTEND.
3. Restore the preserved config.js values if the deployment process overwrote them.
4. Reload the site and allow the v6.8.0 service worker to replace the older cache.

REQUIRED ACCEPTANCE TESTS
A. Alphabetical records
- Add students with names beginning with A, M, and Z.
- Confirm the Student Directory displays them in alphabetical full-name order.
- Add teachers with names beginning with A, M, and Z.
- Confirm the Teacher Directory displays them in alphabetical full-name order.

B. Teacher class visibility
- Assign a class teacher to Basic 3.
- Assign a subject teacher to one subject in Basic 4.
- Log in as each teacher.
- Confirm the class teacher sees current Basic 3 students and officially assigned subject classes only.
- Confirm the subject teacher sees current students only in the assigned subject class.
- Confirm neither teacher sees students whose current placement is outside assigned classes.

C. Promotion placement
- Configure the immediate next academic year and next class level.
- Approve or publish a passing Term 3 report for a Basic 3 student.
- Confirm the student appears in Basic 4 and no longer appears in the active Basic 3 directory.
- Confirm the previous Basic 3 enrolment remains available only in protected history and is inactive.
- Test a student below the cutoff and confirm the student remains in Basic 3.

D. Attendance
- Log in as the Basic 3 class teacher.
- Confirm Attendance appears in navigation.
- Confirm only Basic 3 is available in the attendance class selector.
- Mark and save attendance for a valid school date.
- Reopen the date and confirm the saved statuses remain correct.
- Mark enough dates to verify term totals.
- Open or create a report card and confirm Days Present and Days School Opened are updated automatically.
- Log in as a subject-teacher-only account and confirm Attendance is not available.

E. Backup
- Run a manual encrypted backup after recording attendance.
- Confirm the backup manifest includes class_attendance_registers and student_attendance_entries.

ROLLBACK
- If deployment acceptance fails, restore the v6.7.4 frontend and database backup according to FINAL_BACKUP_AND_RESTORE_RUNBOOK.md.
- Do not delete attendance data manually during rollback without a verified backup.
