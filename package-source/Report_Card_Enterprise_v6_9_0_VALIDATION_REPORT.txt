VALIDATED ENHANCEMENTS
1. Platform role and isolation
- The platform_super_admin enum role is added in a committed transaction and recognized by current_app_role().
- The role receives only the Platform Licensing navigation route.
- Bootstrap data excludes academic years, terms, classes, subjects, and academic permissions for the platform role.
- School System Administrator role selectors do not offer platform_super_admin.
- The legacy direct browser profile-mutation RPC is revoked.
- The admin-user-management Edge Function rejects crafted attempts to update, delete, deactivate, demote, or reset a Platform Super Administrator account.

2. Strong authentication
- Platform licensing RPCs call a shared Platform Super Administrator authorization gate.
- Licensing changes and access-lock actions require authenticator assurance level AAL2.
- Platform profiles are forced to require MFA.
- The protected setup script creates or converts only a separate, unlinked Auth account and synchronizes its Auth role metadata.

3. Licence lifecycle
- Starter, Professional, and Enterprise plans are seeded.
- Issue date, activation date, expiry date, grace-period end, status, reference, compliance reason, and notes are supported.
- Pending activation, active, grace period, expired, suspended, revoked, and perpetual states are supported.
- Active, grace-period, and perpetual licences receive an activation timestamp when one is not supplied.
- Suspended and revoked states require a clear compliance reason.
- Grace-period status requires both expiry and grace-end dates.
- Perpetual status rejects an expiry date.
- Existing installations receive a safe perpetual Enterprise licence when no licence is present.

4. Access control and continuity
- System-Administrator-only, school-wide, and platform-wide lock scopes are supported.
- Read-only and deny-access lock modes are supported.
- Optional automatic lock end times and audited release controls are supported.
- Applicable deny locks take precedence over read-only locks.
- Expired, suspended, and pending-activation licences preserve licensed reads while disabling school writes.
- Revoked licences and applicable deny locks block school portal access.
- Existing records are never automatically deleted because of a plan or licence state.
- Manual encrypted backup access remains available to the System Administrator in read-only mode.

5. Server-side enforcement
- Separate restrictive SELECT, INSERT, UPDATE, and DELETE RLS guards are installed on protected school tables.
- DELETE explicitly requires licence write permission, preventing deletion in read-only mode.
- Core write triggers protect SECURITY DEFINER workflows as well as direct table operations.
- Student and teacher plan capacities are enforced by database triggers.
- System Administrator capacity is enforced by the hardened admin-user-management Edge Function.
- Student photos, report PDFs, Principal signatures, and report-card template Storage policies apply licence read/write checks.
- Existing report-PDF publication authority and System Administrator template authority remain intact.

6. Audit and data protection
- Licence events and verification logs are append-only for browser users.
- Initial licence creation, licence updates, access locks, releases, and platform administrator provisioning are recorded.
- Verification logging is throttled to avoid repeated equivalent records within fifteen minutes.
- Licence-key hashes are removed from portal and event-history responses.
- Internal licence notes are not returned in ordinary school bootstrap data.
- All five licensing tables are included in encrypted scheduled backups.

7. Reusable package integrity

- No 08_schema.sql file was created.
- The browser package generator includes the v6.9.0 SQL, documentation, setup script, Edge Functions, and validation report.
