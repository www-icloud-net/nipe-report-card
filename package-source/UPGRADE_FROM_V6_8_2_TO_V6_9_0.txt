REPORT CARD ENTERPRISE
UPGRADE FROM CONFIRMED v6.8.2 TO v6.9.0

IMPORTANT
Keep the confirmed v6.8.2 ZIP and a verified encrypted backup. Apply this upgrade to a test or cloned Supabase project before production.

DATABASE
1. Open the v6.9.0 package.
2. Open 07_schema.sql.
3. Run only the section beginning:
   REPORT CARD ENTERPRISE v6.9.0
   PLATFORM SUPER ADMINISTRATOR AND LICENSING CONTROL PLANE
4. Confirm the final result:
   07 SCHEMA v6.9.0 PLATFORM LICENSING CONTROL: PASS
5. Do not run 01 through 06 again on an existing v6.8.2 database.
6. 08_schema.sql is not required for this release.

SAFE DEFAULT
The migration creates a perpetual Enterprise licence automatically when no licence exists. This prevents the upgrade from locking or restricting the confirmed production installation.

FRONTEND
1. Preserve the production values in GITHUB_PAGES_FRONTEND/config.js.
2. Replace the deployed frontend files with the v6.9.0 GITHUB_PAGES_FRONTEND contents.
3. Hard-refresh the browser or clear the previous service-worker cache.

EDGE FUNCTIONS
Redeploy:
- admin-user-management
- scheduled-backup
The notification-dispatcher function is unchanged but may also be redeployed for release parity.

PLATFORM SUPER ADMINISTRATOR
1. In Supabase Authentication > Users, create a separate platform-owner account.
2. Confirm the email and assign a strong temporary password.
3. Edit PLATFORM_SUPER_ADMIN_SETUP.sql and replace REPLACE_WITH_PLATFORM_ADMIN_EMAIL.
4. Run the setup file in the SQL Editor.
5. Sign in through the normal application login.
6. Complete mandatory authenticator-app MFA enrolment.
7. Confirm the account opens only Platform Licensing and no academic portal.

ACCEPTANCE TESTS
A. Baseline continuity
- Existing System Administrator, Principal, teachers, and parents can sign in.
- The licence displays Enterprise, Perpetual, and Unlocked.
- Existing report, PDF, template, attendance, backup, and workflow features remain operational.

B. Platform separation
- The Platform Super Administrator sees only Platform Licensing.
- A school System Administrator cannot assign the platform role through Users and Access.
- Platform licensing actions require MFA.

C. Licence lifecycle
- Change the test licence to Active with future expiry and save.
- Set it to Grace Period and confirm warning access remains writable.
- Set it to Expired and confirm school users receive read-only access.
- Restore Active or Perpetual and confirm writes return.

D. Access locks
- Apply System Administrator + Read-only and confirm only the school administrator loses write capability.
- Apply School + Read-only and confirm school users can view but cannot change records.
- Apply School + Deny and confirm school users cannot enter the portal.
- Release each lock and confirm access returns immediately.

E. Capacity enforcement
- On a test plan with finite limits, confirm the server blocks creation or reactivation beyond the student and teacher limits.
- Confirm no existing records are deleted when changing to a smaller plan.

F. Backup and audit
- Run a full encrypted backup and confirm the five licensing tables are included.
- Confirm licence changes and lock actions appear in immutable history.

ROLLBACK
1. Restore the v6.8.2 frontend and Edge Functions only after first setting the licence to Perpetual and releasing all active locks.
2. Database objects added by v6.9.0 may remain without affecting v6.8.2, but a full database rollback should use the verified pre-upgrade backup.
3. Never delete school records to resolve a licence state.
